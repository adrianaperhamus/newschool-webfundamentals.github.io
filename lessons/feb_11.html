Whats and whys of the Command Line

Websites and web applications are made of files and folders filled with code and information, hosted on servers.

Command line is a text interface that takes commands and speaks directly to operating system

There are commands for creating and altering files and folders.

GUI (graphical user interface)

Why do developers use the command line?

  • It's efficient – you can perform direct actions quickly
  • It's precise – we tell the computer how to act
  • Use your mouse less, don't interact with GUI

    What is a path?

    Directions from one place to another, from one file to another

    Absolute path

    Relative path

    Where the heck am I? ("print working directory")

    pwd

    Paths go from least specific to most specific. They're like nesting dolls.

    /Milky_Way/Solar_System/Earth/USA/New_York

    What's around me? ("list directory contents")

    ls

    How do I get to where I want to go? ("change directory")

    cd

    Absolute path (~ root directory)

    cd ~/directory/inner_folder

    Relative path (one dot is current directory, two dots is up a level)

    cd ./lessons
    cd ..

    Go up a level

    cd ..

    Now that we know how to navigate directories, how do we create folders and files?

    Make a directory

    mkdir

    Make a file

    touch

    move (or rename)

    mv file_to_move where_to_move

    Be aware of unsafe commands!

    rm
    sudo

    remove file

    rm file_name

    Those are the basics of what you'll need to create files and move around directories when building websites.

    Git key takeaways

    • What is "version control" and why should you care?
    • What is the difference between git and GitHub?
    • How do I save versions and revisions of my code?
    • How do I send and receive information from GitHub?

    Now that we have all of these files and folders, we need a way to keep track of them and how they change over time.

    Git is a distributed version control system, meaning that it keeps track of revisions over time–it allows you to see, go back to, and even revert changes.

    Git and GitHub are not the same thing. Git is the system, and GitHub is a web-based hosting service for Git repositories.

    GitHub is like these things, but for code

    Git takes snapshots of the changes in your code. These are called "commits."

    To start using git, initialize a repository. Then start making your changes.

    git init

    See what files have changed

    git status

    See exactly what changed in those files

    git diff

    "Stage" and "commit" your changes

    git add files_to_stage
    git commit -m "Message"

    See a "log" of what you have changed

    git log

    So we have our snapshots tracked using git, how do we get them to GitHub?

    GitHub is a "remote" version of your repository

    git remote add origin https://github.com/user/repo.git

    This may be getting confusing, so let's try initializing the project we did together, adding a commit and pushing it up to a remote repository.

    git push origin master
